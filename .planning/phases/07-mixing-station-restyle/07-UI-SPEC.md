# Phase 7 — UI Design Contract

> Visual and interaction contract for applying the v1.1 "Modern Spatial Utility" design system (built in Phase 6) to the existing Mixing Station screens. Generated by gsd-ui-researcher. Pure restyle — zero ViewModel or service changes.
>
> Platform: SwiftUI / iOS 17+ native. shadcn not applicable.
>
> **Upstream baseline:** `.planning/phases/06-design-system-foundation/06-UI-SPEC.md` — all design system primitives (spacing, typography, color tokens, SquircleCard, PillButtonStyle, glassmorphism header) are **inherited unchanged**. Phase 7 only defines net-new visual contracts specific to the Mixing Station.

| Property | Value |

## Component Contracts

### Central Connecting Line (`TimelineSpineView`)

The defining visual of the Vertical Timeline Hybrid layout. A decorative thread that runs vertically through the clip section, passing **behind** SquircleCards and **through** the `+` operator chips.

| Property | Value |
|----------|-------|
| File | `SonicMerge/Features/MixingStation/TimelineSpineView.swift` (NEW) |
| Implementation | `ZStack`-level `Rectangle` inside the clip Section's list row background, rendered in a `.background` layer behind all cards in the section. Because `List` rows cannot share a background across rows, the spine is drawn per-row as a 2pt strip filling each row's full height, positioned at a fixed leading inset. Visually continuous because adjacent rows share the same inset and color. |
| Thickness | **2pt** |
| Color | `Color(uiColor: semantic.accentGlow).opacity(0.35)` — Deep Indigo at 35% alpha. No gradient — flat color. |
| Horizontal inset from leading edge of row | **60pt from the leading edge of the row** (16pt list row inset + 16pt card `md` padding + 28pt to reach the horizontal center of the 100pt × 52pt waveform thumbnail minus half thumbnail width... effective visual: line sits on the waveform-thumbnail vertical centerline). **Exact value:** 60pt measured from row leading edge. |
| Z-order | **Behind** SquircleCard. Drawn via `.background(alignment: .leading) { spineStrip }` on each `VStack` row. Cards fully occlude the line where they overlap; the line is visible in the inter-card gap (controlled by `.padding(.vertical, SonicMergeTheme.Spacing.lg)`). |
| Start (top terminator) | Begins 4pt below the bottom edge of the SEQUENCE section (so there is a small air gap between the SEQUENCE label and the line). |
| End (bottom terminator) | Ends 4pt above the top edge of the `=` `MergeOperatorLabel` circle. The `=` operator visually "caps" the spine. |
| Interaction with `+` `MergeOperatorLabel` | The `+` circle (`MergeOperatorLabel(kind: .plus)`) has an **opaque** fill (`semantic.surfaceBase`, not the current 0.55 translucent) so the line visually terminates at the circle border and resumes on the far side. Effect: the line appears to thread **through** the `+` chip. Implementation: bump `MergeOperatorLabel.Circle().fill(semantic.surfaceElevated)` → `Circle().fill(semantic.surfaceBase)` (opaque). |
| Interaction with GapRow pills | The pill row sits at the same leading inset as the clip cards (16pt row inset + 16pt internal padding). Because pills are compact (44pt tall), the spine 2pt line is **visible behind the pill row** in the vertical gaps above/below each pill button, threading the eye through the transition. No special z-order handling needed — pills render on top, spine visible in surrounding whitespace. |
| Behavior with exactly 1 clip | Spine is **hidden** (not rendered). No other cards to connect. Implemented via `if viewModel.clips.count >= 2 { spineStrip }`. |
| Behavior with 0 clips | Spine is **hidden**. Empty state (`VStack` with "No clips yet" and Import button) renders instead — no timeline, no spine. |
| Accessibility | Fully decorative. `.accessibilityHidden(true)`. VoiceOver never announces the line. |
| `reduceMotion` | No-op (line is static — nothing to reduce). |
| `reduceTransparency` | Color opacity raised from 0.35 to 0.55 for stronger contrast when transparency is reduced. |

**Rationale for drawing per-row (not as a single ZStack overlay over the List):** `List` rows in SwiftUI do not expose a shared-background hook, and rendering an absolute-positioned ZStack overlay over a scrolling List causes the line to drift on scroll. Per-row `.background(alignment: .leading)` is the only approach that keeps the line locked to content while remaining compatible with `.onMove` reordering.

### Mesh Gradient Waveform (`MergeSlotWaveformView` restyle)

Restyle the existing `Canvas`-rendered peaks to use a mesh gradient fill instead of a flat `accentWaveform` color.

#### iOS 18+ implementation

```swift
@available(iOS 18.0, *)
MeshGradient(
    width: 3,
    height: 3,
    points: [
        [0.0, 0.0], [0.5, 0.0], [1.0, 0.0],
        [0.0, 0.5], [0.5, 0.5], [1.0, 0.5],
        [0.0, 1.0], [0.5, 1.0], [1.0, 1.0]
    ],
    colors: [
        deepIndigo, deepIndigo, indigoPurpleMix,
        deepIndigo, indigoPurpleMix, purple,
        indigoPurpleMix, purple, purple
    ]
)
```

Where:
- `deepIndigo` = `Color(uiColor: semantic.accentAction)` = `#5856D6`
- `purple` = `Color(uiColor: semantic.accentGradientEnd)` = `#AF52DE` (NEW token)
- `indigoPurpleMix` = visually blended — produced by the 9-point mesh interpolation, no explicit token

The 3×3 control point grid produces a diagonal Indigo→Purple gradient from top-leading to bottom-trailing.

#### iOS 17 fallback

```swift
LinearGradient(
    colors: [deepIndigo, purple],
    startPoint: .topLeading,
    endPoint: .bottomTrailing
)
```

Same start/end colors, same diagonal direction. Visually acceptable side-by-side — the LinearGradient loses the 2D curvature but retains the brand identity.

#### Availability guard

Lives inside `MergeSlotWaveformView.body` as a computed `@ViewBuilder` property:

```swift
@ViewBuilder
private var waveformGradient: some View {
    if #available(iOS 18.0, *) {
        MeshGradient(...)
    } else {
        LinearGradient(...)
    }
}
```

#### Gradient application to Canvas bars

The existing `Canvas` draws per-bar rectangles. Phase 7 changes the fill approach:

- **Before (current):** `context.fill(Path(rect), with: .color(semantic.accentWaveform.opacity(0.95)))` — flat per-bar color.
- **After (Phase 7):** Use `Canvas` to draw a single composite `Path` containing ALL bar rectangles, then fill that path once with `GraphicsContext.fill(path, with: .style(ShapeStyle))`. The gradient is applied as a single full-canvas fill masked by the composite bar path.

Implementation pattern:
```swift
Canvas { context, size in
    let barPath = Path { path in
        for (i, peak) in peaks.enumerated() { path.addRect(barRect(i, peak, size)) }
    }
    // Mask gradient to bar path
    context.fill(barPath, with: .style(/* gradient wrapped in a ShapeStyle */))
}
```

Since `MeshGradient` does not conform to `ShapeStyle` usable inside `Canvas` contexts directly in iOS 18.0, the effective approach is:

**Final pattern:** Replace the `Canvas` background with a **ZStack**: the gradient (mesh or linear) fills the thumbnail rect, and the `Canvas` draws the bar shapes with a **destination-out** blend to punch the non-bar areas to transparent, revealing the thumbnail background. Equivalent alternative: use `.mask { Canvas { bar shapes } }` to mask the gradient by the bar shapes.

**Chosen approach:**
```swift
waveformGradient
    .mask {
        Canvas { ctx, size in /* draw bar rects in black */ }
    }
```

This is the simplest, iOS 17-compatible pattern and works identically for both `MeshGradient` and `LinearGradient` branches.

#### Thumbnail background layer

Retain the existing `RoundedRectangle(cornerRadius: 10).fill(surfaceBase.opacity(0.85))` backing rect **underneath** the gradient-masked waveform. This provides:
- Light mode: soft off-white well that makes the gradient bars read cleanly.
- Dark mode: near-black well that makes the gradient bars glow.

Full z-stack: `[surfaceBase@0.85 rounded rect] → [gradient-masked bars]`.

#### Dark mode handling

Gradient stops are identical in both modes. Contrast is managed by the backing rect (which IS mode-dependent via `surfaceBase`).

## Drag Micro-Animation Contract

**Requirement:** MIX-05 — "Dragging an audio card produces a visible scale-up and elevated shadow animation for the duration of the drag gesture."

**Gesture source:** System `List.onMove` reorder (inherited from existing implementation). Phase 7 does NOT introduce a custom `LongPressGesture` → `DragGesture` chain — doing so would require restructuring the single-Section/ForEach-with-.onMove invariant that protects against the reorder crash fixed earlier.

**Problem:** System `List.onMove` does not expose a "this row is being dragged" boolean. There is no public API to detect per-row drag state when `editMode == .active`.

**Solution:** Use `@GestureState` via a **row-attached, no-op `DragGesture(minimumDistance: 0)` that runs in parallel** with the system reorder gesture. The row gesture doesn't consume events (`.simultaneously(with:)` style) — it only mirrors "is a finger currently touching this row" into `@GestureState var isDragTouch: Bool`.

```swift
@GestureState private var isDragTouch: Bool = false

SquircleCard(glowEnabled: isDragTouch) { ... }
    .scaleEffect(isDragTouch ? 1.03 : 1.0)
    .animation(reduceMotion ? nil : .spring(response: 0.28, dampingFraction: 0.72), value: isDragTouch)
    .gesture(
        DragGesture(minimumDistance: 0)
            .updating($isDragTouch) { _, state, _ in state = true }
    )
```

**Note on accuracy:** `isDragTouch` becomes `true` on any finger-down on the card, not strictly on reorder-drag. This is an acceptable approximation — user taps the play button or long-presses to drag, and either way the row visually pops. The play button's own `PillButtonStyle` scale animation still fires independently (scale 0.96 on press) and does NOT conflict — the card scales up (1.03), the button inside scales down (0.96), producing a layered effect that feels intentional.

**If this approximation is unacceptable (MIX-05 checker ruling):** Fallback is to bind the whole card scale/shadow to `editMode` being active (permanent "lifted" look while in edit mode). This is less dynamic but unambiguous. Default implementation: the `@GestureState` approach above. Checker can override.

### Animation parameters

| Property | Resting | Dragging | Transition |
|----------|---------|----------|------------|
| Scale | 1.0 | **1.03** | `.spring(response: 0.28, dampingFraction: 0.72)` |
| Shadow color | `Color.black.opacity(0.10)` | `Color(uiColor: semantic.accentGlow).opacity(0.25)` | same spring |
| Shadow radius | 16 | **24** | same spring |
| Shadow y-offset | 6 | **10** | same spring |

All four animated values change together (driven by the `glowEnabled` parameter flip in `SquircleCard`), producing a unified lift.

### Haptics during drag

| Event | Feedback |
|-------|----------|
| Pickup (finger-down on card) | `.sensoryFeedback(.impact(weight: .medium), trigger: isDragTouch)` — fires once on rising edge |
| During reorder (passing another row boundary) | System default (iOS List provides this automatically in `.onMove` reorder) — no explicit addition |
| Drop (finger-up) | `.sensoryFeedback(.impact(weight: .light), trigger: !isDragTouch)` — fires once on falling edge (lighter than pickup) |

### `reduceMotion` fallback

| Behavior | Normal | Reduce Motion |
|----------|--------|---------------|
| Scale animation | `.spring(...)` | `nil` — scale jumps instantly (no easing) |
| Scale amplitude | 1.0 ↔ 1.03 | 1.0 ↔ 1.03 (retained — subtle, not a motion trigger) |
| Shadow transition | spring animated | instant |
| Haptics | fire | fire (haptics are not motion) |

Rationale: a 3% scale change is below the Reduce Motion threshold for "significant movement"; Apple's HIG allows it. Spring easing IS suppressed.

## PillButtonStyle Additions

The existing `PillButtonStyle` (Phase 6) fills a capsule with Deep Indigo, fixed 12pt × 24pt padding, and assumes a text label. Phase 7 needs two variants the current style does not cleanly express:

1. **Icon-only variant** (play button): square 44×44 bounds, no text, 16pt symbol center.
2. **Unselected / outline variant** (gap pills in unselected state): transparent fill, 1pt `accentAction@0.5` stroke, `textPrimary` label.
3. **Compact horizontal padding** (gap pills): 16pt horizontal instead of default 24pt to fit 4 pills in card width.

**Proposed addition to `PillButtonStyle.swift`:**

Add parameters to `PillButtonStyle`:

```swift
struct PillButtonStyle: ButtonStyle {
    enum Variant { case filled, outline }
    enum Size { case regular, compact, icon }

    var variant: Variant = .filled
    var size: Size = .regular

    // ... existing implementation branches on variant/size
}
```

Usage:
```swift
Button { ... } label: { Image(systemName: "play.fill") }
    .buttonStyle(PillButtonStyle(variant: .filled, size: .icon))

Button("0.5s") { ... }
    .buttonStyle(PillButtonStyle(
        variant: isSelected ? .filled : .outline,
        size: .compact
    ))
```

**Exact spec per variant/size matrix:**

| Variant/Size | Fill | Stroke | Label color | H-pad | V-pad | Min-height | Inner glow |
|-------------|------|--------|-------------|-------|-------|-----------|-----------|
| filled / regular | `accentAction` | none | white | 24pt | 12pt | 44pt | yes |
| filled / compact | `accentAction` | none | white | 16pt | 10pt | 44pt | yes |
| filled / icon | `accentAction` | none | white | 0 | 0 (`.frame(width:44, height:44)`) | 44pt | yes |
| outline / regular | clear | 1pt `accentAction@0.5` | `textPrimary` | 24pt | 12pt | 44pt | no |
| outline / compact | clear | 1pt `accentAction@0.5` | `textPrimary` | 16pt | 10pt | 44pt | no |

Haptic (`.impact(weight: .light)` on press) and press-scale (0.96 with spring) behave identically across all variants. `reduceMotion` suppression inherited.

**If the Phase 6 `PillButtonStyle` is not extended:** Phase 7 can implement a local fork as `PillIconButtonStyle` and `PillOutlineButtonStyle` in the same file. Preferred approach is extending the existing style (single source of truth). Planner chooses.

## Implementation Constraints (Hard)

These are NOT design decisions — they are guardrails the planner/executor must respect. A violation blocks the phase.

1. **No ViewModel changes.** `MixingStationViewModel`, `AudioMergerService`, `WaveformService` are frozen. If a visual decision would require a new `@Published` / `@Observable` property, the decision is wrong.
2. **Single Section + `.onMove` invariant.** The clip `ForEach` remains inside one Section. Drag animations use `@GestureState` on the row, never a custom long-press/drag composition that breaks `.onMove`.
3. **iOS 17 minimum.** Every `MeshGradient` reference must be behind `#available(iOS 18.0, *)` with a LinearGradient fallback. Build must succeed on iOS 17 simulator.
4. **Max 2 blur layers per screen** (v1.1 GPU budget, inherited from STATE.md). Mixing Station's blur count: 1 layer (glassmorphism header — Phase 6). Phase 7 adds **0** blur layers. SquircleCard clip cards use `glassEnabled: false`. Do not flip `glassEnabled: true` on clip cards.
5. **No new Phase 6 token modifications.** Existing tokens (`accentAction`, `surfaceCard`, etc.) keep their values. Only `accentGradientEnd` is added (see [New Tokens Required](#new-tokens-required)).
6. **Forbidden font weights.** `.heavy`, `.bold`, `.medium`, `.black` — every occurrence in touched files must be migrated to `.regular` or `.semibold`.
